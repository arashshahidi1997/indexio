# Pluggable RAG Engines for indexio

## Overview

indexio currently hand-rolls its retrieval pipeline (langchain-text-splitters,
langchain-chroma, langchain-huggingface). Instead of maintaining this plumbing,
indexio could delegate to external high-level RAG engines — either a single
default engine or corpus-specific engines chosen for their strengths.

The key constraint: indexio's value is the **config layer and ecosystem contract**
(YAML source registration, multi-store, `sync_owned_sources`), not the retrieval
internals. The engine should be swappable without changing the surface API.

## Candidate Engines

### General-purpose (default engine candidate)

- **txtai** — [github.com/neuml/txtai](https://github.com/neuml/txtai)
  - Closest match to indexio's philosophy: YAML-configurable, lightweight,
    embeddings DB with built-in semantic search
  - Supports hybrid search (sparse + dense), graph networks, SQL queries
  - Handles chunking, embedding, and indexing in one package
  - Would replace all four langchain deps with one

- **FlashRAG** — [github.com/RUC-NLPIR/FlashRAG](https://github.com/RUC-NLPIR/FlashRAG)
  - Research toolkit (EMNLP 2025) with Chunkie chunking library
  - Worth studying for chunking strategies even if not used as engine

### Optimized for document/PDF output (biblio corpus)

- **LightRAG** — [github.com/HKUDS/LightRAG](https://github.com/HKUDS/LightRAG)
  - Graph-augmented retrieval: dual-level (chunk + inter-dependency)
  - Incremental indexing without full rebuild — relevant for growing paper corpora
  - Heavier LLM requirements (32B+ recommended for entity extraction)

- **RAG-Anything** — [github.com/HKUDS/RAG-Anything](https://github.com/HKUDS/RAG-Anything)
  - Multimodal RAG from the same team as LightRAG
  - Could handle docling derivatives with tables/figures better than pure text search

### Optimized for code (codio corpus)

- Code-specific embedding models (e.g., `jinaai/jina-embeddings-v3`,
  `Salesforce/SFR-Embedding-2`) may outperform general-purpose models on
  code search
- Worth evaluating whether codio corpus needs a different embedding strategy
  vs. just a different model within the same engine

### Storage-efficient / local-first

- **LEANN** — [github.com/yichuan-w/LEANN](https://github.com/yichuan-w/LEANN)
  - 97% storage savings via pruned graph + on-demand recomputation
  - Relevant if indexio targets large local corpora

## Possible Architectures

### A. Single engine, swap internals (recommended starting point)

Replace langchain deps with txtai. Keep the same config surface. Simplest path.

```
indexio config (YAML) → txtai Embeddings → search API
```

### B. Engine-per-corpus (future option)

Each corpus can declare its engine in config. indexio routes queries to the
right backend and merges results.

```yaml
sources:
  - id: docs
    corpus: docs
    engine: txtai          # default

  - id: biblio_docling
    corpus: papers
    engine: lightrag       # graph-augmented for papers

  - id: codio-notes
    corpus: codelib
    engine: txtai
    embedding_model: jinaai/jina-embeddings-v3   # code-tuned model
```

This is more complex but lets each corpus use what works best.

### C. Federated query layer (most ambitious)

indexio becomes a query router. Each ecosystem package can bring its own
engine. indexio only owns the config registry and query fan-out/merge.

Risk: leaky abstraction, hard to keep result scores comparable across engines.

## Decision Criteria

- How much does retrieval quality vary across engines for our corpus types?
- Is the complexity of multi-engine worth it, or does a single good engine
  with per-corpus embedding models cover 90% of the need?
- How stable are the candidate engines' APIs? (txtai has been stable for years;
  LightRAG is newer and research-driven)

## Notes

- The ecosystem contract (`sync_owned_sources`, YAML config, `rag_query` API)
  must remain stable regardless of engine choice
- codio, biblio, notio should never need to change when the engine changes
- Start with Strategy A; only move to B if benchmarks show meaningful
  quality gaps across corpus types
